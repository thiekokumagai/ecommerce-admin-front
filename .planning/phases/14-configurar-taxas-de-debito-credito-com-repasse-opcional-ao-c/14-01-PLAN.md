# Plan: Configurar taxas de debito/credito com repasse opcional ao cliente

Este plano descreve as mudanças necessárias no painel administrativo frontend (`ecommerce-admin-front`) para permitir que o lojista escolha se as taxas de cartão de crédito e débito devem ser repassadas ao cliente (adicionadas ao total cobrado) ou absorvidas pela loja (reduzindo apenas o líquido no caixa).

---

## 🎯 Objetivos

1. **Tipagem Flexível**: Atualizar a interface `PaymentRule` para suportar o boolean `passedToCustomer` (Repassar para o cliente).
2. **Formulário de Configurações**: Adicionar uma chave liga/desliga (switch) nas regras de pagamento do tipo "Acréscimo" (`charge`) em `PaymentRulesSettingsForm.tsx`.
3. **Cálculo Condicional no Recebimento**: Modificar o `OrderDetailDrawer.tsx` para que as taxas de crédito/débito só gerem o `cardSurcharge` (aumento no total a pagar) se a regra tiver `passedToCustomer: true`.
4. **Sem Alterações no Backend**: O backend já armazena as regras como JSON flexível (`any`) e calcula a taxa retida da operadora (`cardFee`) com base no `totalReceived` e no percentual da regra. Portanto, as lógicas de backend permanecerão intocadas, pois a ausência do repasse (sem `cardSurcharge`) manterá o `totalReceived` igual, e a taxa do gateway será deduzida normalmente, afetando apenas o saldo líquido no caixa.

---

## 🛠️ Detalhamento das Mudanças no Frontend

### 1. Tipagem das Configurações
- **Arquivo**: `src/types/settings.ts`
- **Ação**: Adicionar `passedToCustomer?: boolean` à interface `PaymentRule`.

### 2. Formulário de Configurações
- **Arquivo**: `src/components/settings/PaymentRulesSettingsForm.tsx`
- **Ação**:
  - Na listagem de regras configuradas, se o `type === "charge"`, renderizar um novo campo Switch "Repassar ao cliente?".
  - Ligar este Switch à função `handleUpdateRule(rule.id, 'passedToCustomer', val)`.
  - Definir que regras recém-criadas tenham `passedToCustomer: true` como padrão (se for charge).

### 3. Recebimento de Pedidos (OrderDetailDrawer)
- **Arquivo**: `src/components/OrderDetailDrawer.tsx`
- **Ação**:
  - No `useEffect` de inicialização das taxas, ao encontrar uma `activeRule` (seja de PIX, Débito ou Crédito), verificar o tipo:
    - Se for `discount`, o repasse não se aplica (o desconto sempre é dado).
    - Se for `charge` e `activeRule.passedToCustomer !== false` (ou seja, repassar), aplicar o cálculo para `cardSurcharge`.
    - Se for `charge` e `activeRule.passedToCustomer === false`, então `initialCardSurcharge = 0`.
  - Refletir a mesma lógica nos métodos `handlePaymentMethodChange` e `handleInstallmentsChange`.
  - Como `cardSurcharge` se torna 0 nesse cenário, o `totalReceived` cobrado do cliente não sobe, mas quando a requisição for enviada para API, a API continuará usando o percentual da regra para calcular a dedução do caixa (taxa de gateway).

---

## ✅ Critérios de Validação

1. **Repasse Ativo**: Criar uma taxa de 5% de crédito repassada. Ao selecionar Cartão no pedido, o total cobrado deve subir. No caixa, deve aparecer a taxa cobrada no bruto e deduzida no gateway.
2. **Repasse Inativo (Absorvido)**: Criar uma taxa de 5% de crédito NÃO repassada. Ao selecionar Cartão no pedido, o total cobrado NÃO deve subir. No caixa, o bruto fica igual e o líquido fica menor (pois a operadora reteve 5%).
3. **Persistência**: Ao recarregar a tela de configurações, os botões de repasse continuam gravados com os valores corretos.
